# Database Migrations

DKPlaylister uses [Alembic](https://alembic.sqlalchemy.org/) for database schema management.

## Setup

Alembic is configured in `alembic.ini` and `alembic/env.py`.

The project uses a `src/` layout, so `alembic/env.py` adds the `src` directory to `sys.path`.

## Workflow for Schema Changes

1. Make your changes to the Pydantic models in `src/dkplaylister/models.py` and the corresponding SQLAlchemy models in `src/dkplaylister/storage.py`.

2. Generate a new migration:
   ```bash
   alembic revision --autogenerate -m "Add new field to songs"
   ```

3. Review the generated file in `alembic/versions/`. **Always review autogenerated migrations** — Alembic is good but not perfect.

4. Test the migration:
   ```bash
   alembic upgrade head
   alembic downgrade -1
   alembic upgrade head
   ```

5. Commit the migration file along with your model changes.

## Running Migrations

- On normal startup (`get_session()`), Alembic will automatically run any pending migrations.
- You can also run manually:
  ```bash
  dkplaylister init          # Recommended for first-time setup
  alembic upgrade head       # Manual upgrade
  alembic current            # Check current revision
  alembic history            # View migration history
  ```

## During Development

If you are iterating quickly and want to reset your local database:

```bash
rm data/dkplaylister.db
dkplaylister init
```

This will create a fresh database and run all migrations from the beginning.

## Legacy Manual Migrations

Older versions of the project used manual `PRAGMA table_info` checks in `storage.py`. These have been converted to proper Alembic migrations and are kept only as a fallback during the transition period.

## Best Practices

- Never edit an existing migration file after it has been merged.
- Prefer small, focused migrations over large ones.
- Always include both `upgrade()` and `downgrade()` methods.
- Document complex migrations in the docstring of the revision file.
